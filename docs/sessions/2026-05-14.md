session. Faces are now readable in dialog, every zone has been densified,
and every NPC has visible idle behaviour beyond bobbing.

## Done

### T1 — Portrait system *(addresses the face complaint)*

- New [`entities/PortraitFactory.ts`](../../src/entities/PortraitFactory.ts):
  composes the head + shoulders crop (top 32 px of LPC frame 130 idle-down)
  of each cosmetic layer at 3× scale into a 96×96 canvas texture. Cached
  by appearance hash on the scene's `TextureManager` — built once per NPC

- [`entities/NPC.ts`](../../src/entities/NPC.ts) exposes `portrait` key;
  `NPCSpec` gains optional `dialog: DialogLine[]` field. If set, `addNPC`
  auto-wires interact → `speakAs(npc, dialog)` so the portrait flows
  through without callers having to.
- [`world/BaseZoneScene.ts`](../../src/world/BaseZoneScene.ts) `openDialog`
  signature picked up an optional `portrait` key; new `speakAs(npc, lines)`
  helper for NPC-sourced dialog.
- [`types/index.ts`](../../src/types/index.ts) `DialogRequest` gained
  optional `portrait` field.
- [`scenes/UIScene.ts`](../../src/scenes/UIScene.ts) dialog box height
  bumped 64 → 80 to fit a 64×64 portrait with padding. Portrait sprite +
  inset frame rect render on the left when a dialog has one; text
  position + word-wrap width recompute around the portrait.
- [`scenes/TownScene.ts`](../../src/scenes/TownScene.ts) and
  [`scenes/EngineeringDistrictScene.ts`](../../src/scenes/EngineeringDistrictScene.ts)
  switched the existing dialog NPCs (greeter + work-history foremen) to
  the new `dialog` field — they get portraits for free.

## Decisions

- **Portrait textures live on the scene's `TextureManager`** rather than a
  module-level cache. Re-entering a scene rebuilds them on demand, which
  is fine because composition is fast (≈1 ms per portrait).
- **`NPCSpec.dialog` field instead of forcing every caller to use
  `speakAs`** — most NPCs have a static dialog, and an explicit
  `onInteract` would have meant every dialog NPC repeating
  `() => this.speakAs(...)` boilerplate. The `addNPC` helper now does the
  right thing automatically.
- **Procedural rather than atlas-sourced dungeon floor** — the atlas's
  slate variants all have grass tufts (designed for outdoor paths). A
  4×4 procedural flagstone with grout reads cleaner for an indoor
  dungeon.
- **`makeBuildingTex` parameterised by colour rather than two separate
  procedural methods** — keeps the recipe in one place and lets us
  cheaply add a third or fourth building variant later.

### T5 — Stone↔dirt auto-tile (Engineering avenue)

- New `makeStoneDirtAutoTile()` in
  [`scenes/PreloadScene.ts`](../../src/scenes/PreloadScene.ts) generates
  a 9-tile set (`tex_sd_nw`, `_n`, `_ne`, `_w`, `_c`, `_e`, `_sw`, `_s`,

- [`scenes/EngineeringDistrictScene.ts`](../../src/scenes/EngineeringDistrictScene.ts)

  stone instead of hard-cutting.

### T6 — Day/night tint overlay

- [`world/BaseZoneScene.ts`](../../src/world/BaseZoneScene.ts) now owns a
  `dayNightOverlay` field plus `refreshDayNight()`. A 5-anchor color
  cycle (dawn `#ff9a5c·0.18` → day `transparent` → dusk
  `#5c4a8c·0.20` → night `#0b1830·0.55` → predawn `#3a2a4c·0.32`) runs
  on an 8-minute loop driven by `Date.now()`, with linear interpolation
  between anchors. Drawn at `Z.Overlay+10`, scroll-locked, refreshed
  every 200 ms.
- Picks up automatically in every scene that extends `BaseZoneScene` —
  no per-zone wiring.

### T7 — Trinkets UI (surfacing chest discoveries)

- New [`scenes/TrinketsUIScene.ts`](../../src/scenes/TrinketsUIScene.ts):
  full-screen modal listing every entry in `STATIC_CHESTS`. Opened
  chests show `flavorTitle + reward + flavorBody` in gold; unopened
  show `◇ undiscovered • hidden in <humanZone(scene)>` muted. Daily
  secrets counted separately in the header so the static-vs-rotating
  split is visible.
- W/S (and arrows) scroll the 6-row viewport via an `offset` field;
  Esc closes. Refresh on every scroll re-reads the save so the list
  stays live with concurrent chest opens (defensive, but cheap).
- Settings hint updated to `[T] view trinkets   [R] reset save   [Esc] close`
  and `keydown-T` launches the new scene.

### T8 — First-launch onboarding overlay

- New [`scenes/IntroUIScene.ts`](../../src/scenes/IntroUIScene.ts):
  full-screen overlay shown only when `save.flags.hasSeenIntro` is
  unset. Names the project, lists controls (WASD/E/Esc), enumerates
  the six zones with a one-line description each, and flags
  `src/data/portfolio.ts` as the one file to edit for stale content.
  Dismissed with Enter/Space/Esc — sets the flag so returning players
  never see it again. Settings → reset save re-arms it.
- [`scenes/TitleScene.ts`](../../src/scenes/TitleScene.ts) `start()`
  intercepts: if `IntroUIScene.shouldShow()` is true, launches the
  overlay and re-calls `start()` after dismissal. Second call sails
  past the check because the flag is now set. Single-keypress flow
  for the player.

## Decisions (T5–T8)

- **Procedural stone↔dirt overlay rather than atlas crop** — keeps the
  set self-contained and matches the existing palette without picking
  through LPC's slate variants (which have grass tufts).
- **Day/night driven by wall-clock `Date.now()` not session time** —
  cheap "feels alive" effect that costs no state and survives
  scene transitions for free.
- **Trinkets read live from the save on every scroll** rather than
  caching a snapshot — chest opens are rare and the save cache is
  already in-memory, so cost is negligible and avoids stale views.
- **Intro flag stored under `save.flags.hasSeenIntro`** — uses the
  existing flags bucket rather than a new top-level save field, so no
  schema bump needed.
