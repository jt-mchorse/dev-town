## Decisions

- **Portrait textures live on the scene's `TextureManager`** rather than a
  module-level cache. Re-entering a scene rebuilds them on demand, which
  is fine because composition is fast (≈1 ms per portrait).
- **`NPCSpec.dialog` field instead of forcing every caller to use
  `speakAs`** — most NPCs have a static dialog, and an explicit
  `onInteract` would have meant every dialog NPC repeating
  `() => this.speakAs(...)` boilerplate. The `addNPC` helper now does the
  right thing automatically.
- **Procedural rather than atlas-sourced dungeon floor** — the atlas's
  slate variants all have grass tufts (designed for outdoor paths). A
  4×4 procedural flagstone with grout reads cleaner for an indoor
  dungeon.
- **`makeBuildingTex` parameterised by colour rather than two separate
  procedural methods** — keeps the recipe in one place and lets us
  cheaply add a third or fourth building variant later.

## Decisions (T5–T8)

- **Procedural stone↔dirt overlay rather than atlas crop** — keeps the
  set self-contained and matches the existing palette without picking
  through LPC's slate variants (which have grass tufts).
- **Day/night driven by wall-clock `Date.now()` not session time** —
  cheap "feels alive" effect that costs no state and survives
  scene transitions for free.
- **Trinkets read live from the save on every scroll** rather than
  caching a snapshot — chest opens are rare and the save cache is
  already in-memory, so cost is negligible and avoids stale views.
- **Intro flag stored under `save.flags.hasSeenIntro`** — uses the
  existing flags bucket rather than a new top-level save field, so no
  schema bump needed.

### T9 — In-world face overlay

The structural problem: LPC sprites are 64×64, with facial features
occupying ~2 px each. At the 480×270 game canvas they collapse to noise.
Earlier the portrait system (T1) solved this in dialog; T9 solves it
in-world.

- New [`entities/FaceFactory.ts`](../../src/entities/FaceFactory.ts):
  builds a 4-frame canvas spritesheet (`tex_face_overlay`, 256×64) with
  one frame per Direction. Down: two 3×3 eyes with a 1-px white
  catchlight + 4-px mouth with two smile-curl pixels. Left/right:
  single profile eye + short mouth. Up: blank back-of-head. Frames are
  registered as named sub-frames so `setFrame(0..3)` works.
- [`entities/LPCCharacter.ts`](../../src/entities/LPCCharacter.ts) adds
  a `faceSprite` field, builds it after every cosmetic layer, and sets
  its depth to 3.5 — between shirt (3) and hair (4) — so hair fringe
  and hats occlude naturally. Direction syncing wired into both
  `walk()` and `faceDirection()` so the face turns with the body.
- [`entities/PortraitFactory.ts`](../../src/entities/PortraitFactory.ts)
  draws hair + hats *after* compositing face features at 3× scale
  (9×9 eyes, 12×3 mouth + 3×3 smile dimples), so the dialog portrait
  carries the same expression as the in-world sprite.
- Verified via preview: zoomed snapshot of the player's head clearly
  shows two black eyes with catchlights below the hair fringe.

### T10 — Concave-corner auto-tile

- [`scenes/PreloadScene.ts`](../../src/scenes/PreloadScene.ts) extends
  the procedural stone-dirt generator with 4 concave variants
  (`tex_sd_c{nw,ne,sw,se}`) — each paints a `STRIP×STRIP` stone wedge
  in one corner only, representing the inner corner where two dirt
  regions touch diagonally.
- [`world/BaseZoneScene.ts`](../../src/world/BaseZoneScene.ts) adds
  `AutoTileSet` interface (concave keys optional) and a new cell-mask
  painter `paintRegionAutoTile(cellSet, set, interior?, mask?)` that
  inspects all 8 neighbors per cell — interior cells with a missing
  diagonal pick the relevant concave variant; sets without concave
  variants fall back to plain interior.
- `rectsToCellSet(rects[])` helper turns one or more rects into the
  cell-key set the painter expects.

### T11 — Brick-path auto-tile

- [`scenes/PreloadScene.ts`](../../src/scenes/PreloadScene.ts) ships a
  new procedural brick set: 16 tiles painted with a running-bond brick
  body + grass overlays on every "out" side, plus strips (`hs`/`vs`),
  end-caps (`capN/S/E/W`), and an isolated variant (`iso`).
- `AutoTileSet` extended with optional strip/cap/iso keys; the painter
  handles 4-cardinal-out (iso), 3-out (cap), 2-opposite-out (strip),
  2-adjacent-out (corner), and 1-out (edge) cases in turn.
- [`scenes/TownScene.ts`](../../src/scenes/TownScene.ts) replaces the
  three flat brick rects + manual edge strips with a single
  `paintRegionAutoTile` call. A `membershipMask` includes the plaza
  cells so brick cells that abut the plaza don't paint grass against
  stone; cells *inside* the plaza paint plain interior brick over the
  existing stone-dirt auto-tile so the path crosses through cleanly.

## Decisions (T9–T11)

- **Bigger eyes over LPC fidelity.** 3×3 black blocks on the head break
  with LPC's tiny native facial pixels. That's the point — readability
  beats canonical proportions.
- **Catchlight pixel.** A single white pixel at the upper-left of each
  eye gives the face a "looking at you" read at gameplay scale.
- **Depth 3.5 for face.** Between shirt (3) and hair (4) so hair fringe
  and hats occlude naturally. Avoids fighting for layer order.
- **Cell-mask painter, not a generic 47-tile Wang engine.** Captures
  the cases the project actually triggers (interior + 4 edges + 4
  corners + 2 strips + 4 caps + iso + 4 concaves) without the cost of
  the full 47-tile set. Sets ship only the keys they need; missing
  keys fall back to the closest base tile.
- **`membershipMask` parameter.** The cleanest way to say "these cells
  exist for neighbor-checks but don't paint them" — needed wherever
  one auto-tile region crosses into another.
