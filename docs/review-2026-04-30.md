## What's wrong (catalogued)

### Critical — the face problem (user's repeated complaint)

LPC body sprites have **~8-pixel-tall faces on a 64-pixel character** (≈12 %
of the character's vertical pixels). At the current `charScale: 1.0` the face
occupies 12–15 native canvas pixels, which is 24–30 displayed pixels at 2×.
You can see *that* there's a face there; you cannot read expression or
distinguish NPCs by features.

**Why scaling alone doesn't fix it:**
- `charScale: 1.5` → character is 3 tiles tall, breaks tile alignment, camera shows ~half the world per frame.
- `charScale: 2.0` → 4 tiles tall, badly out of scale with environment.
- Camera zoom in → fewer tiles visible per frame; world feels small.

**The standard pixel-RPG solution: portraits during dialog.** Stardew Valley,
Eastward, Coromon — all show a 64×64 or 96×96 character portrait in the
dialog box next to the speech text. The overworld stays compact; the moment
that matters (talking to someone) gets a face. This is the right fix.

### High — terrain rough edges remain

| Where | What's wrong |
|---|---|
| Engineering avenue (north/south of sand shoulders) | stone meets sand at a hard rectangle — no auto-tile available for stone↔sand |
| Cottage / farm foundations | dirt patch under each meets surrounding grass at a hard rect (foundations don't use the auto-tile painter) |
| Brick path ends at world edges | brick just stops, no end-cap tile |

The auto-tile painter we have only handles grass↔dirt 3×3. To fix all
terrain pairs we'd need to cut additional sets from the atlas (or paint
procedural ones for stone↔sand which the atlas doesn't ship).

### Medium — environment density / readability

- **AI/ML Lab**: server racks not visible in default camera view; red carpet runway is a thin strip that reads as a stripe, not a feature. Zone reads as "stone room with three NPCs."
- **Full-Stack Dungeon**: corridors feel empty; crates and rubble piles are 16-pixel sprites lost in the negative space. Floor uses the procedural `tex_wall` for the wall sections, which is the placeholder, not real atlas art.
- **Engineering**: trees lining the path are all the same oak variant. Buildings vary by foreman appearance, but the buildings themselves are identical 128×128 procedural tiles. The third building's label sometimes clips at the viewport edge depending on player position.
- **Skill Grove totems**: procedural stone pillars look out of place on a dirt clearing. A wooden or clay totem would fit the earth theme better.

### Low — UI / polish

- **HUD**: just a credits readout in the corner. No mini-map, no compass, no current-zone label, no NPC-nearby indicator other than `[E]` floating over interactables.
- **News kiosk**: dialog wraps long headlines awkwardly because the dialog box is fixed-width and headlines vary.
- **Fishing UI**: text-only feedback (no animated bobber). Functional but less satisfying than what Stardew does.
- **Modal hint text**: now ASCII (W/S, A/D, E, Esc) — fine.

## Diagnosis: why iterating hasn't fixed the "world looks ok but not great"

We've been making one-off fixes in response to specific complaints. The
underlying gaps are *structural*:

1. **No portrait system** — small overworld characters are intrinsic to LPC.
   You can't fix this with another `charScale` bump.
2. **Auto-tile system only handles one terrain pair** — every other
   transition (stone↔sand, stone↔dirt, sand↔water beyond the top row,
   brick↔grass for path-ends) is a hard cut and we patch with
   strip-tiles ad-hoc.
3. **Procedural fallback art lives alongside LPC art** — looks immediately
   "off" because they're different art styles colliding. Cottages,
   fences, totems, walls, kiosks are all procedural.
4. **No per-NPC visual variety** — each foreman is randomly outfitted but
   otherwise indistinguishable; buildings are identical.

Tackling those four root causes will move the project from "playable
prototype" to "polished pixel-art game" without further per-symptom patching.

## Plan

Four tiers in priority order. Each tier is a single focused session, not a
multi-day project.

### Tier 1 — Portrait system *(addresses the face complaint)*

**Scope:** add a portrait slot to the dialog UI, render per-character
portraits by composing LPC layers at 4× zoom (head + shoulders region only).

1. New `entities/PortraitFactory.ts`: takes an `Appearance`, renders 96×96 PNG
   to an in-memory canvas by drawing each LPC layer's frame-0 (idle-down)
   top-quarter at 4× scale.
2. Cache portrait by appearance hash; build once per NPC at scene-init.
3. Extend `DialogRequest` with optional `portrait: HTMLCanvasElement | string`.
4. Update `UIScene` to allocate a 96×96 slot on the left of the dialog box
   when a portrait is provided; reflow text accordingly.
5. Wire NPCs: `BaseZoneScene.openDialog` already takes a speaker; refactor to
   take an NPC ref and pass the cached portrait.

**Effort:** 2–3 h. Single PR. Biggest user-visible win.

### Tier 2 — Auto-tile system for all visible terrain pairs

**Scope:** generalise the auto-tile painter beyond grass↔dirt; cut the
additional transition sets that exist in the atlas; procedurally generate
the rest.

1. Extract the LPC slate-path 5×5 set (rows 15–19, cols 0–4) for stone↔grass.
2. Try the LPC pond 3×3 set (rows 11–13, cols 9–11) — we already cut top
   shore tiles for fishing; expand to full set for general water edges.
3. Procedural stone↔sand and sand↔dirt — generate via the same Phaser
   canvas pipeline used for `makeTile`, with edge gradients.
4. Refactor `paintRectWithAutoTileBorder` to accept *any* 9-set + an
   inside-corner variant (for concave joins where two auto-tile rects touch).
5. Apply to: Engineering avenue edges, Town cottage / farm foundations,
   Fishing Dock full water shoreline (right/left/bottom edges, not just top).

**Effort:** 3–4 h. Probably needs eyeballing in browser per tile to confirm.

### Tier 3 — Environment density / per-NPC variety

**Scope:** make zones feel populated and individual NPCs distinct.

1. **AI/ML Lab**: drop a second row of monitor sprites; widen the red carpet
   to 3 tiles; add a whiteboard sprite (procedural) on the back wall.
2. **Dungeon**: bigger crate piles (cluster 3–4 crates at a corner), more
   rubble, add cobweb sprites at corridor corners. Floor tile to a darker
   stone variant.
3. **Engineering**: alternate two building textures (current procedural +
   a second variant with different wall colour); add a street lamp sprite
   between buildings.
4. **Per-NPC portraits make every foreman / shopkeeper / quiz NPC visually
   distinct in dialog** — Tier 1 unlocks this for free.

**Effort:** 2–3 h.

### Tier 4 — Later polish

- Concave-corner auto-tile.
- NPC idle behaviours beyond bobbing (occasional wave, look-around).
- Day/night lighting overlay using `Phaser.GameObjects.Layer` with a tinted
  rectangle that follows a `Date.now()`-driven cycle.
- Animated fishing bobber (now possible if we ship the bobber as a regular
  decoration sprite in the FishingDockScene rather than a UI overlay).

**Effort:** open-ended.
